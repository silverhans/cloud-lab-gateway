# Task B1.2 — Moodle emulator with LTI 1.3 launch

You are a coding agent working on **Cloud Lab Gateway**. Read `CLAUDE.md` and `docs/ARCHITECTURE.md` first.

## Goal

A standalone Go service (`cmd/moodle-emulator`) that simulates Moodle for end-to-end testing and the demo. It serves an HTML page listing fake courses with "Запустить лабу" buttons. Clicking → POSTs a properly signed LTI 1.3 `id_token` to the gateway's `/lti/launch`. Also serves `/jwks.json` so the gateway can verify the signature.

This is the **bridge that unblocks B2.1 (LTI verifier)** — they can be developed in parallel because they communicate over a clean protocol.

## Why this matters

- Live demo needs Moodle, and a real Moodle instance is heavy. The emulator is a 5-file Go binary.
- LTI 1.3 is hard to test without a generator. The emulator IS the test fixture for the verifier.
- For evaluation: criterion 6 (Moodle integration) requires a working sign-launch flow. The emulator is the proof that the gateway accepts LTI 1.3 launches correctly.

## Files to create

```
cmd/moodle-emulator/
  main.go                # HTTP server, routing, JWT signing
  templates/
    index.html           # course list + launch form
    courses.go           # static course/lab data

  claims.go              # struct types reused by gateway verifier and emulator
```

The new `internal/lti/` package is shared between this emulator and the future gateway-side verifier (B2.1). Define **only** the claims schema (structs + JSON tags + Zod-equivalent validation), no signing or verification — those are adapter-specific.

## LTI 1.3 launch — what to generate

The id_token is a JWT signed RS256 with the following claims:

```json
{
  "iss": "https://moodle-emulator.local",
  "aud": "cloud-lab-gateway",
  "sub": "user-{external_id}",                // stable per-emulator user
  "exp": <now + 5min>,
  "iat": <now>,
  "nonce": "<32-byte hex>",
  "azp": "cloud-lab-gateway",
  "https://purl.imsglobal.org/spec/lti/claim/deployment_id": "emu-deployment-1",
  "https://purl.imsglobal.org/spec/lti/claim/message_type": "LtiResourceLinkRequest",
  "https://purl.imsglobal.org/spec/lti/claim/version": "1.3.0",
  "https://purl.imsglobal.org/spec/lti/claim/roles": [
    "http://purl.imsglobal.org/vocab/lis/v2/membership#Learner"
  ],
  "https://purl.imsglobal.org/spec/lti/claim/context": {
    "id": "<course-external-id>",
    "label": "linux-101",
    "title": "Linux основы"
  },
  "https://purl.imsglobal.org/spec/lti/claim/resource_link": {
    "id": "<lab-template-slug>",
    "title": "Лабораторная работа №1"
  },
  "https://purl.imsglobal.org/spec/lti/claim/custom": {
    "lab_template_slug": "linux-basics-1"
  },
  "email": "<external_id>@emulator.local",
  "name": "Student <external_id>"
}
```

Sign with a static RSA-2048 keypair, generated **once at process startup** and persisted to `/tmp/moodle-emulator-key.pem` (or read from `EMULATOR_PRIVATE_KEY_PEM_PATH` if set — useful for tests that need a stable key). Expose the public half at `/jwks.json` in the [JWKS](https://datatracker.ietf.org/doc/html/rfc7517) format with `kid` matching the JWT header's `kid` claim (use `EMULATOR_LTI_KID` env, default `emu-key-1`).

## HTML page (templates/index.html)

Simple Go html/template. Server-rendered (no JS framework). Just enough to be presentable:

- Header: "Moodle Emulator (demo)"
- Pick-a-user dropdown (5 fake students + 1 teacher)
- Two course cards, each with 2-3 lab buttons:
  - "Linux основы" → labs: `linux-basics-1`, `linux-basics-2`
  - "Nginx и конфиги" → labs: `nginx-config-1`
- Each button submits a form: `POST /launch` (emulator), which:
  1. Builds the id_token with the chosen user + lab + course
  2. Renders an auto-submit form that POSTs to `{EMULATOR_GATEWAY_URL}/lti/launch` with `id_token`, `state`, fields

The auto-submit form lets the browser carry the launch directly to the gateway — exactly how real Moodle does it.

Style: minimal CSS embedded in the template. Goal is functional + readable, not pretty.

## Acceptance criteria

1. `go build ./cmd/moodle-emulator` produces a working binary.
2. `go test -race ./cmd/moodle-emulator/...` passes.
3. `docker compose -f deployments/docker-compose.yml --profile demo up moodle-emulator` boots the service; visiting `http://localhost:9000` lists courses; clicking a Launch button issues a POST to the gateway and the gateway logs the launch.
4. `/jwks.json` returns valid JWKS that can verify the JWT (proven by the lti_test.go roundtrip).
5. JWT claims match the LTI 1.3 spec for `LtiResourceLinkRequest` (verify against IMS spec linked below).
6. No hard-coded private key in source — generated on startup or loaded from env path.
7. `gitleaks detect --no-git --source=.` clean.
