# Task B3.1 — Frontend: API integration, real auth, student flow

You are a coding agent on **Cloud Lab Gateway**. Read `CLAUDE.md` and
`docs/ARCHITECTURE.md` first. **Your scope is `web/` only** — do not touch Go code.

## Goal

The frontend skeleton (B1.1) is built — Vite/React/TS/Tailwind/shadcn, TanStack
Router + Query, four role pages — but **every page renders mock data**. This
task turns the student experience into a real application:

1. Wire the **real, typed API client** (the generated `api.gen.ts` is not yet
   connected).
2. A query/mutation hook layer with centralized RFC-7807 error handling.
3. **Real session auth** — `/auth/me`, login, logout — replacing the
   localStorage mock.
4. The **student page** wired end-to-end to real endpoints + live SSE.

This task establishes the integration pattern that **B3.2** (teacher/admin)
follows — get it clean.

## Current state — study these first

- `web/src/lib/api.ts` — an `openapi-fetch` client typed with a **stub**
  `FallbackPaths = Record<string, never>`, plus an ad-hoc `apiFetch<T>` helper.
  Both are placeholders.
- `web/src/lib/api.gen.ts` — **generated** from `api/openapi.yaml` via
  `npm run gen:api` (`openapi-typescript`). 1100+ lines, accurate to the
  contract, currently unused.
- `web/src/lib/auth.ts` — a localStorage demo-user mock (`getStoredUser`,
  `setDemoUser`, `clearStoredUser`). **Note:** `login-page.tsx` imports
  `demoAuthEnabled` and `moodleEmulatorUrl` from here, which do not exist yet —
  the skeleton has loose ends; make the project typecheck cleanly.
- `web/src/lib/types.ts` — hand-written UI types (`LabInstance`, `CheckRun`, …)
  that are camelCase and **do not match** the OpenAPI snake_case schema.
- `web/src/lib/sse.ts` — a Zod discriminated union for SSE events
  (`lab.state_changed`, `lab.deploy_progress`, …) — provisional shapes.
- `web/src/hooks/use-current-user.ts` — tries the localStorage mock first, then
  `GET /api/v1/auth/me`.
- `web/src/hooks/use-labs-stream.ts` — opens `EventSource('/sse/labs')`.
- `web/src/pages/student-page.tsx` — a complete, attractive layout (stepper,
  access card, TTL card, checks, quota) but bound to `demoLab` from
  `mock-data.ts`; **no button is wired**.
- `web/src/router.tsx` — TanStack Router (code-based), `/login` + an
  authenticated layout with `/student /teacher /admin`.
- `api/openapi.yaml` — the contract (source of truth). `LabInstance`,
  `LabInstanceDetail`, `CheckRun`, `Problem` schemas; `Problem` is RFC-7807.

## Step 1 — real API client (lib/api.ts)

```ts
import createClient from "openapi-fetch";
import type { paths } from "./api.gen";

export const api = createClient<paths>({
  baseUrl: "/api/v1",
  credentials: "include",
});
```

Re-run `npm run gen:api` so `api.gen.ts` is fresh. Delete `FallbackPaths` and
the ad-hoc `apiFetch` helper — everything goes through the typed client.

## Step 2 — hook layer + error handling

- `web/src/lib/problem.ts` — openapi-fetch returns `{ data, error }`; `error`
  is the RFC-7807 `Problem`. One helper normalizes it: extracts `code`,
  `title`, `detail`, fires a `sonner` toast, and returns a typed error the
  caller can branch on (e.g. `ERR_QUOTA_EXCEEDED`, `ERR_POOL_EMPTY`,
  `ERR_LAB_ALREADY_ACTIVE`).
- `web/src/hooks/` — typed TanStack Query hooks, one per endpoint the student
  flow needs: `useLabs`, `useLab(id)`, `useCreateLab`, `useStopLab`,
  `useFreezeLab`, `useRunCheck`, `useLabChecks`. Mutations invalidate the
  relevant `['labs', …]` query keys.

## Step 3 — types: generated as source of truth

Replace the hand-written shapes in `types.ts`. Re-export the generated schema
types — `type LabInstance = components["schemas"]["LabInstance"]` etc. Where a
component needs derived/formatted values (a TTL %, a display title), add a thin
**view-model mapper** function — never a parallel hand-maintained type. No `any`
outside `api.gen.ts`; `tsc` strict must stay clean.

## Step 4 — real auth (auth.ts, use-current-user.ts, login-page.tsx)

- `useCurrentUser` → `GET /api/v1/auth/me` only (drop the localStorage-first
  path). 401 → `null` (anonymous).
- `login-page.tsx`: the **primary** path is the Moodle-emulator link (students
  enter via LTI). Add the missing `auth.ts` exports — `moodleEmulatorUrl()`
  (from `import.meta.env.VITE_MOODLE_URL`) and `demoAuthEnabled()` (from
  `import.meta.env.VITE_DEMO_AUTH === "true"`). Keep the email/password form
  calling `POST /api/v1/auth/login`; on RFC-7807 errors show inline messages.
- Keep the **dev-mode role switch** (`setDemoUser`) behind `demoAuthEnabled()`
  as a development/demo fallback, clearly labelled "dev".
- Logout → `POST /api/v1/auth/logout`, then invalidate `['auth','me']`.

> **Dependency:** a real `clg_session` cookie is issued by the backend (LTI
> launch, or a later dev-login endpoint). `POST /auth/login` may return `501`
> until the backend identity task lands — handle that gracefully; the dev-mode
> switch remains the working fallback meanwhile.

## Step 5 — student page wired to real data

`StudentPage`:

- Fetch the student's active lab via `GET /labs`; if there is one, load detail
  via `GET /labs/{id}` (`LabInstanceDetail` — has `ki_resources`,
  `deploy_steps`, `latest_check_run`).
- **No active lab** → a "Запросить стенд" call-to-action → `POST /labs`
  (`course_id` / `lab_template_id` from `import.meta.env` for the demo).
  `ERR_QUOTA_EXCEEDED` / `ERR_POOL_EMPTY` / `ERR_LAB_ALREADY_ACTIVE` → a clear
  message, not a generic toast.
- Wire every button: "Завершить стенд" → `DELETE /labs/{id}`; "Сообщить о
  проблеме" → `POST /labs/{id}/freeze` (with a reason); "Запустить проверку" →
  `POST /labs/{id}/checks`; checks list → `GET /labs/{id}/checks`.
- "Скачать SSH-ключ" → `GET /labs/{id}/ssh-key` returns
  `application/x-pem-file` (a text body, **not** JSON) — fetch as text, trigger
  a browser download (Blob + anchor, filename `lab-<id>-key.pem`).
- Drive the lifecycle stepper from the real `state` / `deploy_steps`, not the
  hard-coded `completedSteps` heuristic.

## Step 6 — live SSE

Keep `useLabsStream` on `/sse/labs`. On `lab.deploy_progress` /
`lab.state_changed`, update the lab query cache so the stepper animates live.
The `sse.ts` Zod schema is **provisional** — the backend SSE broker is the
authoritative contract; keep parsing tolerant (`safeParse`, ignore unknown
event types) so a schema drift never crashes the page.

## Backend reality — contract-first

Only `POST /labs` is fully live; other endpoints may still return `501` while
the backend use cases are being built. The typed client is correct against the
contract — wire everything now; a `501`/`ERR_NOT_IMPLEMENTED` must render a
friendly "функция скоро будет" state, never a crash. Pages light up the moment
the backend endpoint goes live.

## Acceptance criteria

1. `npm run gen:api` runs cleanly; `npm run typecheck`, `npm run lint`,
   `npm run build` all pass with **no warnings**.
2. No `any` outside `api.gen.ts`. TypeScript strict stays on.
3. `FallbackPaths` and the ad-hoc `apiFetch` are gone; one typed client.
4. Loading + error states on every async boundary; every error is a toast or
   inline message derived from the RFC-7807 `Problem`.
5. The student page renders entirely from API data — `demoLab` is no longer
   imported by it.
6. Anonymous → `/login`; the role switch / login routes to the right page.
7. The build serves through `deployments/Dockerfile.frontend` + nginx unchanged.
