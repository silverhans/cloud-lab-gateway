# Task B1.3 — HTTP API foundation (chi mux, RFC-7807 errors, session auth, POST /labs)

You are a coding agent on **Cloud Lab Gateway**, a Go backend. Read `CLAUDE.md`
and `docs/ARCHITECTURE.md` first.

## Goal

Stand up the HTTP API package `internal/adapters/httpapi/` — the foundation the
whole REST surface plugs into. **Most use cases do not exist yet** (only
`lab.CreateLab` is implemented), so this task delivers the *skeleton plus the one
wireable endpoint*, done to production quality:

1. Generate handlers + types from `api/openapi.yaml` (`make gen-openapi`).
2. A chi mux with the middleware stack: session-cookie JWT auth + a centralized
   RFC-7807 error renderer.
3. Implement `POST /labs` end-to-end against `lab.CreateLab`.
4. Every other operation: a uniform `501 Not Implemented` Problem response — a
   clear, typed placeholder that later use-case tasks replace.
5. Wire the mux into `cmd/gateway/main.go`.

The point is to **lock the contract, the error format, and the auth seam** so the
frontend (B1.1) can integrate against a real server and later tasks just fill in
handler bodies.

## Read first (do not modify)

- `api/openapi.yaml` — the contract, source of truth. 19 operations.
- `api/oapi-codegen.yaml` — generator config: **strict-server**, package
  `httpapi`, output `internal/adapters/httpapi/generated.go`.
- `internal/domain/shared/errors.go` — sentinel errors + `ErrorCode()`. The
  error mapper is built on these.
- `internal/app/lab/create.go` + `deps.go` —
  `CreateLab(ctx, CreateInput) (*labdomain.LabInstance, error)`; `CreateInput`

  use-case bundle.
- `cmd/gateway/main.go` — process bootstrap; the mount seam is the commented
  `router.Mount("/api/v1/", httpapi.NewMux(deps))` line. Note the existing chi
  middleware (`RequestID`, `RealIP`, `Recoverer`, `zapRequestLogger`) and that
  `/healthz` + `/readyz` are already served at root by `mountHealth`.

## Step 1 — generate

The generated code imports `github.com/oapi-codegen/runtime` — add it to
`go.mod`. Run `make gen-openapi`, commit `internal/adapters/httpapi/generated.go`.
With `strict-server: true` you implement a `StrictServerInterface` — typed
request and response objects per operation. The spec has **no `operationId`s**,
so oapi-codegen derives method names from method+path (`PostLabs`, `GetLabsId`,
`PostLabsIdFreeze`, …). **Implement exactly the methods the generated
`StrictServerInterface` declares** — the names in this brief are indicative.

## Step 2 — files to create

```
internal/adapters/httpapi/
  deps.go            # Deps — collaborators the handlers need
  mux.go             # NewMux(deps) http.Handler — chi router + generated handler + middleware
  server.go          # the StrictServerInterface implementation type
  errors.go          # error → RFC-7807 Problem mapping
  auth.go            # session-cookie JWT: validation middleware + IssueSession + principal-in-context
  labs.go            # PostLabs (real) + the other lab operations (501)
  health.go          # GetHealthz / GetReadyz — trivial, to satisfy the generated interface
  notimplemented.go  # errNotImplemented sentinel + the remaining operations
  *_test.go          # handler tests with a fake lab use case (net/http/httptest, no DB)
```

(The file split is a suggestion. What matters: one `StrictServerInterface`
implementation, every operation present, no business logic in handlers.)

## Step 3 — error mapping (errors.go)

One function maps any `error` to an HTTP status + a `Problem` body, rendered as
`application/problem+json` (schema `Problem` in the spec). Use
`shared.ErrorCode(err)` for the stable `code` field. Status table:

| error | status |
|---|---|
| `ErrInvalidInput`, body decode / validation failure | 400 |
| `ErrUnauthorized` | 401 |
| `ErrForbidden` | 403 |
| `ErrNotFound` | 404 |
| `ErrLabAlreadyActive`, `ErrPoolEmpty`, `ErrQuotaExceeded`, `ErrIdempotencyClash`, `ErrInvalidTransition` | 409 |
| `ErrCloudUnavailable`, `ErrLMSUnavailable` | 503 |
| `errNotImplemented` (package-local sentinel) | 501 |
| anything else | 500 |

For 500s: log the real error with the request id (zap), but the `Problem`
`detail` returned to the client must be generic — never leak internals. Set
`Problem.instance` to the request id (`chi/middleware.GetReqID`).

## Step 4 — auth (auth.go)

Session = `clg_session` HttpOnly cookie, a JWT signed **HS256** with
`CLG_JWT_SECRET`. Use `github.com/golang-jwt/jwt/v5` (add to `go.mod`).

- `Principal` type: `{UserID uuid.UUID; Role string; CourseRoles map[uuid.UUID]string}`.
  Carried in `context.Context`; provide `principalFrom(ctx) (Principal, bool)`
  and an unexported context key.
- `RequireAuth` middleware: parse + verify the cookie JWT; on failure render a
  401 Problem; on success put the `Principal` in context. Apply it to all
  `/api/v1/*` routes **except** those the spec marks `security: []`
  (`POST /auth/login`). Health is served at root by `main.go` — not your concern.
- `IssueSession(w http.ResponseWriter, p Principal, ttl time.Duration)` — mints
  the JWT and sets the cookie: `HttpOnly`, `SameSite=Lax`, `Path=/`, `Secure`
  when not in dev mode. **This task owns the cookie format and validation; it
  does not own credential checking.** `POST /auth/login` and the LTI launch
  (which will call `IssueSession`) are a later identity task — for now
  `PostAuthLogin` returns 501.
- `GetAuthMe` reads the `Principal` from context → `UserMe` (only the fields
  available without a UserRepo: `id`, `role`; `display_name` may be empty).
- `PostAuthLogout` clears the cookie, returns 204.

JWT claims: `sub` = user id, custom `role`, `exp`, `iat`.

## Step 5 — PostLabs (the one real endpoint)

`POST /api/v1/labs` → `lab.CreateLab`:

1. Decode the body (`course_id`, `lab_template_id`, both UUID); a malformed body
   → `ErrInvalidInput` (400).
2. `StudentUserID` comes from the authed `Principal` — **never from the body**.
3. `RequestID` = `chi/middleware.GetReqID(ctx)`.
4. Call the use case; map the `*labdomain.LabInstance` result to the
   `LabInstance` response model; respond `201`.
5. Errors flow through the mapper — `ErrQuotaExceeded` / `ErrPoolEmpty` /
   `ErrLabAlreadyActive` become 409 Problems with their stable codes, exactly as
   the spec's `POST /labs` description promises.

Define a narrow interface so handlers are unit-testable with a fake:

```go
type LabService interface {
    CreateLab(ctx context.Context, in lab.CreateInput) (*labdomain.LabInstance, error)
}
```

`httpapi.Deps` holds a `LabService` (satisfied by `lab.Deps`), a `*zap.Logger`,
and a dev-mode bool.

## Step 6 — wire into main.go

Replace the commented seam with `router.Mount("/api/v1", httpapi.NewMux(deps))`.
The full `lab.Deps` graph (repos, UoW, queue…) may not be constructible in
`main.go` yet — if so, leave a clearly-marked `TODO` and a minimal construction
that compiles. **`go build ./cmd/...` must stay green.**

## The 501 operations

Every operation except `PostLabs`, `GetHealthz`, `GetReadyz`, `GetAuthMe`,
`PostAuthLogout` returns the `errNotImplemented` sentinel through the error
mapper (→ 501 Problem). Write **one** helper, not 19 copies. `GetStreamLabs`
(SSE) and `PostAuthLogin` are explicitly later tasks.

## Tests

Table-driven handler tests with `net/http/httptest` and a **fake `LabService`**
— no database, no Redis. Cover:

- `POST /labs` happy path → 201, body is the lab JSON, `student_user_id` equals
  the principal's id (not the body's).
- `POST /labs`, fake returns `ErrQuotaExceeded` → 409,
  `code: ERR_QUOTA_EXCEEDED`, content-type `application/problem+json`.
- `POST /labs` with no / an invalid session cookie → 401.
- `POST /labs` with a malformed JSON body → 400.
- A 501 endpoint (e.g. `GET /labs`) → 501 Problem.
- `IssueSession` then a request carrying that cookie → `RequireAuth` passes; a
  tampered cookie → 401.

## Acceptance criteria

1. `make gen-openapi` regenerates cleanly; `generated.go` committed.
2. `go build ./...` passes, including `cmd/gateway`.
3. `go test -race -count=1 ./internal/adapters/httpapi/...` passes.
4. `gofmt -l -s .`, `staticcheck ./...`, `go vet ./...`, `gosec ./...` clean.
5. Compile-time check: `var _ StrictServerInterface = (*Server)(nil)`.
6. Every error response is `application/problem+json` with at least
   `type`, `title`, `status`, `code`.
7. Handlers contain no business logic (CLAUDE.md rule 3): decode → call use
   case → render.
